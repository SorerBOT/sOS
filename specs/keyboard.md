# Keyboard Spec
This spec details the requirements of any keyboard driver implemented in the project, be it a ps2 driver, a USB driver, or whatever.  

## The General Idea

### Single Instance Driver
The simplest possible keyboard driver listens to keyboard inputs, interprets their meaning (i.e. transforms a key-id into a concrete character), and keeps track of those events, so that it can serve them to the OS on demand.

### Abstracting the "Single Instance Driver" -- Attempt 1
To my initial comprehension, the keyboard driver's role was to, on demand, return the character corresponding to the key currently pressed. Only later did it strike me that a keyboard driver has no way whatsoever of establishing that relation, as it is completely oblivious to the type of keyboard its operator is using; be it an English keyboard equipped with the QWERTY layout, or a Hebrew keyboard equipped with the SI-1452-1 layout. Distinguishing different languages and different layouts seems to be out of the driver's scope, and so we must resolve to a lower mutual criteria. My initial idea was to create an internal set of "key-id"'s, and have the OS translate these key-ids based on the current language/layout. The issue with this approach is that it is "lossy" in the sense that it completely revokes the structural aspect of the keyboard; which consists of neighbouring, and general key positions. For instance, when the user presses "control + c" in order to copy something, the ascii code for 'c' is completely irrelevant. We don't care in the slightest whether the current language & layout combination identify this character as 'c', or the Hebrew 'ב' or whatever, we simply care about a certain position being clicked on the keyboard. The same goes for the WASD keys in video games, where the video games care not about 'A' being pressed, but rather about a key in a certain position being held down. For this reason, the prevalent way to identify keys is by their physical positions. To my understanding, most keyboards have the same set of physical keys whose interpretation differs based on the user's preference (handled by the OS). With this mindset, the driver's role would be to simply highlight which keys are currently pressed, which keys have just been released and so on. The role of transforming this into an actual state with resolved control-keys and characters would be delegated to some part of the kernel, or a libc extension.

### Why Attempt 1 is Disasterous
Well... it all sounds good on paper. We enjoy the advantages of knowing the exact position of the key being pressed / released, and we completely abstract away the selected language / layout. Unfortunately, we've skimmed over a tiny detail which I will forever remember with the saying "the devil is in the detail" engraved onto it, and that detail, is that keyboards come in different shapes; two examples, are the ANSI 104-key keyboard, popular in America, and the ISO 105-key keyboard popular in Europe. Both keyboards have the same keys, with some, like the ENTER and SHIFT keys appearing in slightly different orientations (taking up a different number of rows / columns), and some which are only available on one type of keyboards (specifically the key to the right of the left SHIFT key on ISO keyboards).

## Concrete Specs
Every keyboard has some internal format to identify keys. For example, PS/2 keyboards use some scancode set (1, 2, or 3), USB keyboards use HID, and there can be numerous more formats. The first role of any keyboard driver, is to resolve this format into the unified format described in the introductory sections which consists of the key's physical position on the keyboard. The second and last role of the keyboard driver is to have this event (keycode + pressed / released) recorded into a buffer managed by the OS itself.  
It will later be the role of the OS managed keyboard handler, to, when queried by the OS, iterate over all the recorded keyboard events and transform each event into either an ASCII character being written down, or an action-sequence taking place. This handler, is supposed to maintain the state of the keyboard's control keys (control, shift, alt, capslock, etc), and update this state as it reads keyboard events. When it encounters a non-control key, the handler should decide based on the current state of the control keys, whether or not it is an action sequence, or an ASCII character being written down, then couple this information and return it to the OS in the form of a "keyboard events unit".

### Building a Keyboard Events Unit
As I already wrote, the keyboard handler should iterate over the buffer of keyboard events, transforming groups of events into "Keyboard Events Units". Iterating over the buffer, the handler decides that it has found an entire unit if it reaches one of these terminal events:
* An ASCII convertable key
* An action key / key combination.

ASCII convertable keys are straightforward, however action keys / key combinations might get tricky; a single action key is something like F1, it is not convertable to ASCII, and is designated to indicate some action. A combination, is something like control + a, or alt + shift + s, which is basically a key paired with an action control key or a bunch of them. In this way, we're treating control + a as a single unit, which is not translatable into ASCII. It has several advantages; firstly, the reader needs no effort to determine the intention of the unit, as it is not translatable to ASCII, and is hence strictly an action unit, and secondly, the sequence can work no matter the current keyboard layout; meaning that even if the currently selected language / layout, control + a is initiated by pressing the same keys. This is also useful in the structural usecase, exemplified by the WASD keys in a video game.
